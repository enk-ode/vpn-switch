## Combinator Architecture

`vpn-switch` treats commands as **rewrite rules**, not imperative procedures.
Each invocation performs one rewrite step:

- a **terminal** function rewrites to shell code (normal form),
- a **combinator** rewrites to exactly one `vpn-switch` command,
- a **batch combinator** rewrites to N `vpn-switch` commands.

Four properties follow from this:

1. **Functions are pure transformers** — read args, write to stdout, no side effects.
2. **Dispatch resolves a command line to a function name** by concatenating
   arguments with underscores until a name in the metadata table matches.
3. **Interpreters control execution** — a per-function interpreter decides how a
   function's output is processed (executed, displayed, logged).
4. **Combinators chain automatically** — non-terminal output is fed back through
   dispatch.

### Why this design

| Requirement | How it is met |
|-------------|---------------|
| Testability | Source the script, call a function directly, assert on its stdout. No mocks. |
| Dry-run / inspection | Set any interpreter to `cat` to see the next rewrite step without executing it. |
| Extensibility | Add a protocol by adding `_<proto>_*` functions and regenerating metadata; dispatch is unchanged. |
| Maintainability | Strict separation: dispatch resolves names, functions generate the next step, interpreters execute. |
| Simplicity | One POSIX-shell script plus `include/*.sh` modules; no runtime dependencies. |

### Inspection in practice

Because the default terminal interpreter is `cat`, a command shows what it would
do before doing it:

```bash
vpn-switch wireguard import config.conf
# cp "config.conf" "$VPN_SWITCH_BASE/wireguard/config.conf"
# chmod 0400 "$VPN_SWITCH_BASE/wireguard/config.conf"

vpn-switch wireguard import config.conf | sh   # actually run it
```

Setting `VPN_SWITCH_INTERPRETER_<name>` to `cat` lets you step through a
combinator chain one rewrite at a time.

### Rejected alternatives

| Approach | Why rejected |
|----------|--------------|
| Monolithic `case` statements | Hard to test, debugging needs code edits. |
| One script per command | Distribution and environment-sharing complexity. |
| State machine | Overkill for the domain. |
| External config DSL | Adds a dependency, reduces transparency. |

## Core Components

### Anchor functions

Named `_<protocol>_<operation><arity>`. The arity suffix is the argument count,
so `wireguard start privacy` (one argument) resolves to `_wireguard_start1`.

```bash
__wireguard_start1() {     # combinator: outputs one vpn-switch command
  local category="$1"
  # resolve category → config, allocate session id
  echo "\"\$VPN_SWITCH_CONTEXT_SCRIPT\" wireguard connect 12345"
}

_wireguard_connect1() {    # terminal: outputs shell commands
  cat <<EOF
wg-quick up "$session_dir/wg0.conf"
pfctl -f /etc/pf.conf
EOF
}
```

### Metadata-driven dispatch

The full list of anchor functions lives in the build-time variable
`ANCHOR_FUNCTIONS`, and a function→module map in `FUNCTION_MODULES`
(`vpn-switch.sh:11,14`, regenerated by `gmake metadata`). `to_function_call()`
(`vpn-switch.sh:510`) filters that list by prefix and matches the longest
underscore-joined name; `process_arguments()` (`vpn-switch.sh:2849`) then loads
the owning module on demand and runs the function via `dispatch()`
(`vpn-switch.sh:621`). There are no `case` statements to maintain — adding a
command means adding a function and regenerating metadata.

## Function Classification

Classification is **intrinsic** to the name: the leading-underscore count
determines the default interpreter (`lookup_interpreter`, `vpn-switch.sh:692`).

| Type | Prefix | Output | Default interpreter |
|------|--------|--------|---------------------|
| Terminal | `_` | shell commands | `VPN_SWITCH_TERMINAL_INTERPRETER` (default `cat`) |
| Combinator | `__` | one `vpn-switch` command | `VPN_SWITCH_COMBINATOR_INTERPRETER` |
| Batch combinator | `___` | N `vpn-switch` commands | `VPN_SWITCH_BATCH_COMBINATOR_INTERPRETER` |

The three types are **disjoint** — a single function never mixes shell commands
with `vpn-switch` commands. If a problem does not fit one type, split it.

### Terminal functions: generation time vs execution time

A terminal function runs at **generation time** and emits concrete commands for
**execution time**. Do the smart work (loops, file checks, command-availability
probes) while generating; emit unrolled, inspectable commands.

```bash
# Loop runs NOW; output is concrete per-file commands
_inspect_configs0() {
  for config in "$VPN_SWITCH_BASE/wireguard/"*.conf; do
    [ -f "$config" ] || continue
    echo "echo '  Found: $(basename "$config")'"
  done
}
# Output:
#   echo '  Found: us-01.conf'
#   echo '  Found: uk-12.conf'
```

The wrong version emits the loop itself, which is opaque in a trace.

For **non-modifying inspection**, a terminal function may run the probe at
generation time and emit the result as comment lines (prefix `# `) so a
misconfigured interpreter cannot execute it:

```bash
_system_inspect0() {
  uname_result=$(uname -a 2>/dev/null | sed 's/^/# /')
  printf '# ## System Info\n%s\n' "$uname_result"
}
```

### Variable addressing

- `$CMD_*` — platform-specific commands, expanded at generation time from
  `template/platform/`.
- `$VPN_SWITCH_*` — user-configurable binaries, expanded at execution time from
  `.env/`.
- Bare `cat`, `echo`, `ls`, `sed` — standard utilities only.

### Batch combinator output rules

Every non-blank, non-comment line MUST:

1. start with `"$VPN_SWITCH_CONTEXT_SCRIPT"` (kept literal as `\$` in the
   function source so it expands only at execution time),
2. contain exactly one `vpn-switch` command — no `;`, `&&`, `||`,
3. route through the dispatcher — no direct shell commands.

Comments (`#`) and blank lines are allowed and ignored. Error text must be a
comment (`# [ERROR] ...`). These rules are validated by
`vpn-switch-architecture-test.sh`, which exercises each combinator and batch
combinator on success and error paths.

Keeping the literal `$VPN_SWITCH_CONTEXT_SCRIPT` makes batch output portable: it
resolves to whatever script is running, regardless of install location.

## Interpreter System

An interpreter is the shell command a function's stdout is piped to.
`lookup_interpreter()` resolves it from the **function name only** (not its
arguments), most specific first:

1. arity-specific override — `VPN_SWITCH_INTERPRETER_wireguard_add2`
2. family override (arity stripped) — `VPN_SWITCH_INTERPRETER_wireguard_add`
3. type default by underscore count — `VPN_SWITCH_{TERMINAL,COMBINATOR,BATCH_COMBINATOR}_INTERPRETER`

There is no per-argument level (e.g. no `..._wireguard_start_privacy`).

### The three defaults

These live in `template/environment/` and are installed into `.env/default/`.

**Terminal** (`VPN_SWITCH_TERMINAL_INTERPRETER`): `cat` — safe by default,
inspect before executing. Opt into execution with
`echo sh > .env/local/VPN_SWITCH_TERMINAL_INTERPRETER`.

**Combinator** (`VPN_SWITCH_COMBINATOR_INTERPRETER`):

```sh
head -n1 | xargs sh -c '[ "$0" = "--" ] || set -- "$0" "$@"; eval "p=$1"; shift; exec "$p" "$@"' --
```

`head -n1` enforces the one-line contract; `xargs` rebuilds arguments preserving
quoting; the `sh -c` wrapper handles the FreeBSD-`sh`/Debian-`dash` difference in
how `$0` is bound, then `eval`-expands `$VPN_SWITCH_CONTEXT_SCRIPT` and `exec`s
the resulting command (no extra fork). The script path is part of the function
*output*, not the interpreter, so a function chooses self-reference vs delegation.

**Batch combinator** (`VPN_SWITCH_BATCH_COMBINATOR_INTERPRETER`):

```sh
sh -c 'tmp=$(mktemp "$VPN_SWITCH_BASE/.tmp/batch.XXXXXX"); chmod 0600 "$tmp"; cat > "$tmp"; exec "$VPN_SWITCH_CONTEXT_SCRIPT" batch "$tmp" true'
```

It captures all output to a 0600 temp file and delegates to the `batch` command,
which dispatches each line independently (see below). The final `true` is the
cleanup flag.

### Why not `sh` for everything

A single `sh` interpreter would lose:

- **Safety** — terminals would execute on sight, no inspection step.
- **Recursion** — a combinator emits a *string* like
  `"$VPN_SWITCH_CONTEXT_SCRIPT" wireguard connect 12345`; `sh` would try to run
  that literal string as a command. The combinator interpreter instead re-enters
  dispatch.
- **Line-by-line batch handling** — `sh` would run all batch lines in one
  context; the batch interpreter routes each line through dispatch separately.

The interpreter signature therefore also *defines* what output is valid: a
combinator emitting two lines silently loses the second (`head -n1`); a terminal
emitting a `vpn-switch` command shows it as a string rather than executing it.

### Heterogeneous families

A "family" is the set of functions sharing a prefix across arities (e.g.
`wireguard add`). When members have different types it is **heterogeneous**:

```
wireguard add1  → _wireguard_add1    (terminal)
wireguard add2  → __wireguard_add2   (combinator)   ← different type
wireguard add3  → _wireguard_add3    (terminal)
```

A **family** override on a heterogeneous family breaks the odd member out:
`VPN_SWITCH_INTERPRETER_wireguard_add=sh` makes `add2` print its command string
instead of recursing. For such families use **arity-specific** overrides only.
Detect them with:

```bash
grep -o '_*wireguard_add[0-9]' vpn-switch.sh include/*.sh | sort -u
```

Homogeneous families (all one type) are safe to override at the family level.

## Dispatch and Chaining

`vpn-switch wireguard start privacy`:

```
process_arguments wireguard start privacy
  → to_function_call → "__wireguard_start1 'privacy'"   (combinator)
  → run __wireguard_start1, output: "$VPN_SWITCH_CONTEXT_SCRIPT" wireguard connect 12345
  → combinator interpreter re-enters dispatch
    → "_wireguard_connect1 '12345'"                      (terminal)
    → output: wg-quick up ...; pfctl -f ...
    → terminal interpreter (cat by default; sh to run)
```

A batch combinator (`vpn-switch dump`):

```
process_arguments dump
  → ___dump0 (batch combinator) outputs:
      "$VPN_SWITCH_CONTEXT_SCRIPT" wireguard dump
      "$VPN_SWITCH_CONTEXT_SCRIPT" session dump
  → batch interpreter writes them to .tmp/batch.XXXXXX, runs: batch <tmp> true
  → _batch2 reads the file line by line, skips #/blank, validates the
    "$VPN_SWITCH_CONTEXT_SCRIPT" prefix, and dispatches each line via
    process_arguments() — which may itself resolve to another batch combinator.
```

Re-entry uses the in-process `process_arguments()`, not a new `vpn-switch`
process. Exit codes from producer and consumer stages are combined by
`combine_exit_codes()` (`vpn-switch.sh:2722`), which also propagates batch
completion codes (≥128) so failures inside a batch surface to the caller.

### The `batch` command

`batch` is itself a normal command family in `include/database.sh`:

- `___batch0` (no args) — used internally,
- `__batch1 <file>` — validates the file, delegates to `batch <file> false`,
- `_batch2 <file> <cleanup>` (`database.sh:805`) — the terminal that reads the
  file, dispatches each line, tracks success/failure counts, and supports
  fail-fast (default) or keep-going (`VPN_SWITCH_BATCH_KEEP_GOING=1`).

## Cascading Batch Pattern

Complex multi-step operations decompose into a tree of batch combinators that
delegate to more specific subcommands. `init` is the canonical example:

```
init minimal
  → ___init1 minimal (batch)        database.sh:443
      "$SCRIPT" database init
      "$SCRIPT" environment init minimal
      "$SCRIPT" phases sync
  → _database_init0 (terminal)      mkdir/chmod/ln -s
  → _environment_init1 minimal      cp templates into .env/default
  → ___phases_sync0 (batch)         phase orchestration
      → _phases_copy0 (terminal)    copy phase templates into .include/phase
      → ___phases_sync1 firewall    one "phases lift firewall:<backend>" per backend
        → __phases_lift1 firewall:pf (combinator)
          → cat input.sh | "$SCRIPT" lift firewall > output.sh
            └ _lift1 (terminal filter) wraps plain shell into a terminal function
```

Benefits: each subcommand is independently runnable
(`vpn-switch phases lift firewall:pf`), structure is discovered at generation
time, and no function ever sources `vpn-switch.sh` — delegation is via
`"$SCRIPT" subcommand`, preserving environment isolation.

Use this pattern for multi-step operations whose sub-steps are independently
useful and statically known at generation time; avoid it for simple single-step
work or decisions that can only be made at runtime.

### Pure filter functions

`_lift1` (`database.sh:78`) is a stdin→stdout filter: it reads a plain shell
phase template, escapes runtime variables (`$?`, `$@`, `$*`, `$1`…), and wraps
the body into a terminal-function heredoc. Because it is a pure filter it
composes in pipes and is trivial to test (feed stdin, assert stdout).

## Environment Isolation

Every stage runs in a clean environment via `env -` (`run_env()`,
`vpn-switch.sh:366`). Only variables from `.env/default` and `.env/local` are
re-introduced, plus the explicitly-passed `VPN_SWITCH_CONTEXT_SCRIPT` and
`VPN_SWITCH_BASE`:

```sh
env - PATH="$PATH" $env_args ... VPN_SWITCH_CONTEXT_SCRIPT="$script_path" \
  VPN_SWITCH_BASE="$VPN_SWITCH_BASE" "$@"      # vpn-switch.sh:473
```

Consequences: no state leaks between invocations; identical inputs give identical
outputs; functions can be sourced and called in isolation for testing; and a user
override file (`VPN_SWITCH_FUNCTION_OVERRIDE`) can be sourced after the core
script to replace specific functions.

## Phases and the Patch Step

Two mechanisms decouple *what a VPN config says* from *what this machine must do*.

### The patch step — normalising provider configs

Provider configs are written for Linux. A WireGuard `.conf` from a provider
embeds Linux-only DNS hooks:

```
PostUp   = resolvconf -a %i -m 0 -x
PostDown = resolvconf -d %i
```

Running that verbatim is wrong on FreeBSD and hard-codes one DNS strategy. So
`start` never runs the imported file directly. A terminal patch function
(`_wireguard_patch1/2`, `_openvpn_patch1/2`) first produces a **patched session
copy**: Linux DNS hooks are stripped, and for OpenVPN the up/down scripts,
`daemon` mode and `dev` interface are injected. DNS and firewall are then applied
by *phases*, not by the config. One downloaded config thus becomes portable
across OSes, and the patch is inspectable as data via `vpn-switch <proto> patch
<config>` before any connection.

### Phases — the configurable action list

A connect is an ordered list of **phases**, each a generated script fragment.
Defaults (`template/environment/`):

```
VPN_SWITCH_PHASES_CONNECT    = firewall vpn dns
VPN_SWITCH_PHASES_DISCONNECT = dns vpn firewall    # reverse: tear down safely
```

Each phase is rendered from a **backend** — the concrete tool for that step:

| Phase    | Backends (`template/phase/base/`)             |
|----------|-----------------------------------------------|
| firewall | pf, ipfw, iptables, none                      |
| dns      | resolvconf, unbound, djbdns, dnsmasq, none    |
| vpn      | protocol-specific (wg-quick / openvpn)        |

A backend is chosen with `phase:backend` syntax; a bare name uses the database
default and `:none` skips the step. At `phases sync` time
(`resolve_phase_backend`, `vpn-switch.sh:1820`) the chosen template is **lifted**
into the per-session `connect.sh` / `disconnect.sh`, with context variables
(`$dns_server`, `$remote_ips`, `$interface`, `$session_dir`, `$config_file`)
expanded. Add a backend by copying `template/phase/base/user-template.sh`.

### Why this matters

This is the decide-vs-do split applied to OS integration: the combinator core
decides *which* phases and *which* backends, and the generated terminal scripts
are the only code that touches the firewall or resolver. Swapping `firewall:pf`
for `firewall:iptables`, or adding a backend, changes behaviour without touching
dispatch — the leaves carry platform knowledge, the tree stays platform-agnostic.

## PATH Architecture

`vpn-switch` keeps two PATH environments:

| Variable | Used by | Typical value | Purpose |
|----------|---------|---------------|---------|
| `PATH` | `vpn-switch.sh` | `/sbin:/bin:/usr/bin:/usr/local/bin` | Read-only management tools (ifconfig, pfctl, netstat, wg, ps). |
| `VPN_SWITCH_PATH` | generated `connect.sh` / `disconnect.sh` | `/sbin:/bin:/usr/sbin:/usr/bin:/usr/local/sbin:/usr/local/bin` | Privileged bring-up tools (pfctl, wg-quick, openvpn, route). |

Management runs with the narrower PATH (least privilege); the wider sbin set is
baked only into the generated scripts. Both are configured in
`$VPN_SWITCH_BASE/.env/{default,local}/`.

`run_env()` uses `PATH`, not `VPN_SWITCH_PATH` (`vpn-switch.sh:473`); using the
latter would break bootstrap (it is not set before `.env` exists) and grant
unnecessary privilege to management operations.

Example — privileged connect via sudo:

```bash
vpn-switch setenv VPN_SWITCH_INTERPRETER_wireguard_connect "sudo sh"
# vpn-switch.sh finds 'sudo' via PATH; connect.sh finds pfctl/wg-quick via VPN_SWITCH_PATH
```

## Trade-offs

**Strengths**

- Composability — pure transformers chain and test in isolation.
- Debuggability — dry-run the whole system by setting interpreters to `cat`.
- Extensibility — add protocols/commands by adding `_<proto>_*` functions; no
  central dispatch to edit.
- Single-file deployment with `include/*.sh` modules; POSIX-only.

**Costs**

- Indirection — understanding dispatch → interpreter → function takes time.
- Process spawning per rewrite stage; negligible against VPN/network latency,
  and re-entry is in-process via `process_arguments()`.

**Good fit when** there are multiple protocols with similar interfaces, dry-run
inspection matters, frequent extension is expected, and single-file deployment is
preferred. **Poor fit** for a simple single-purpose script or a hot loop.

For `vpn-switch` it fits: two protocols (WireGuard, OpenVPN) with a shared
interface, security-sensitive operations where inspect-before-execute is
valuable, user-extensible firewall/DNS backends, and POSIX-shell single-file
distribution.
