# DNS Troubleshooting Guide

Companion to [DEBUGGING_GUIDE.md](DEBUGGING_GUIDE.md). Covers DNS resolution
failures that can occur after the VPN tunnel itself is up. Most examples assume
FreeBSD with `local_unbound`; the Linux notes at the end cover the equivalents.

How vpn-switch configures DNS (default `resolvconf` backend): the connect DNS
phase pipes `nameserver <dns_server>` into `resolvconf -x -a <iface>.inet`, then
reloads `local_unbound` so it picks up the new forwarder. The DNS server is the
per-protocol default (`VPN_SWITCH_DNS_wireguard`, default `10.2.0.1`;
`VPN_SWITCH_DNS_openvpn`, default `10.96.0.1`) unless overridden via `setenv`.
See `template/phase/{wireguard,openvpn}/connect/dns.sh`.

---

## Problem: DNS Resolution Fails Despite a Working Tunnel

### Symptoms

- Tunnel connects (`wg-quick up` / `openvpn` succeed)
- Ping to the VPN gateway works (e.g. `ping 10.2.0.1`)
- Name resolution fails: `ping example.com` -> "Name could not be resolved"
- A direct query to the VPN DNS works: `drill @10.2.0.1 example.com` succeeds
- The local resolver fails: `drill @127.0.0.1 example.com` -> SERVFAIL

The split between "direct query works" and "local resolver fails" is the key
signal: the tunnel and the VPN DNS are fine, so the fault is in the local
resolver (`local_unbound` on FreeBSD).

### Diagnosis

1. Confirm the tunnel and gateway:
   ```bash
   ifconfig wg0      # or tun0 for OpenVPN
   ping 10.2.0.1     # VPN gateway
   ```

2. Query the VPN DNS directly (should return NOERROR):
   ```bash
   drill @10.2.0.1 example.com
   ```

3. Query the local resolver (SERVFAIL points at local_unbound):
   ```bash
   drill @127.0.0.1 example.com
   ```

4. Check what the resolver was told to do. On FreeBSD, the resolvconf
   subscriber writes the forwarder into `/var/unbound/forward.conf` and points
   the system at the local resolver:
   ```bash
   cat /var/unbound/forward.conf   # expect: forward-addr: 10.2.0.1
   cat /etc/resolv.conf            # expect: nameserver 127.0.0.1
   ```

5. Watch the queries on the wire:
   ```bash
   sudo tcpdump -i wg0 -n port 53   # one terminal
   drill @127.0.0.1 example.com     # another terminal
   ```

### Common Root Cause: DNSSEC Validation Against a Non-Validating Upstream

If the local resolver has DNSSEC validation enabled but the VPN DNS does not
serve the records validation needs (`DNSKEY`, `DS`), every lookup ends in
SERVFAIL even though the A record was returned. The sequence in tcpdump:

```
10.2.0.2.28624 > 10.2.0.1.53: A? example.com.      <- query
10.2.0.1.53 > 10.2.0.2.28624: A 203.0.113.10       <- answer returned
10.2.0.2.32580 > 10.2.0.1.53: DNSKEY? .            <- validation attempt
10.2.0.1.53 > 10.2.0.2.32580: <NXDomain/REFUSED>   <- validation cannot complete
```

The resolver receives a usable A record but cannot validate the chain of trust,
so it discards the answer and returns SERVFAIL.

### Fix: Disable DNSSEC Validation on the Local Resolver

This is a change to *your* resolver configuration, not to vpn-switch. On
FreeBSD `local_unbound`, edit `/etc/unbound/unbound.conf` (back it up first):

```diff
 server:
     username: unbound
     directory: /var/unbound
     chroot: /var/unbound
     pidfile: /var/run/local_unbound.pid
-    auto-trust-anchor-file: /var/unbound/root.key
+    # DNSSEC disabled for VPN-DNS compatibility (VPN DNS does not serve DNSKEY/DS)
+    # auto-trust-anchor-file: /var/unbound/root.key
+    module-config: "iterator"
     interface: 127.0.0.1
```

- Commenting out `auto-trust-anchor-file` removes the trust anchor.
- `module-config: "iterator"` drops the validator module from the pipeline.

Apply:

```bash
sudo cp /etc/unbound/unbound.conf /etc/unbound/unbound.conf.bak
sudo $EDITOR /etc/unbound/unbound.conf   # make the changes above
sudo service local_unbound restart       # restart, not reload, for module-config changes
```

### Verification

```bash
vpn-switch start <config>           # bring the tunnel up
drill @127.0.0.1 example.com        # expect NOERROR with an address
ping example.com                    # should resolve and reply
curl https://ifconfig.me            # should show the VPN exit IP
```

### Alternatives to Disabling DNSSEC

- **Different upstream DNS.** Point the resolver at a DNSSEC-capable resolver
  (e.g. 1.1.1.1, 9.9.9.9). Trade-off: those queries leave the tunnel and leak
  your real DNS traffic, defeating much of the point of the VPN.
- **Per-domain negative trust anchors.** Keep validation on globally and exempt
  specific zones with `domain-insecure`. Brittle in practice, since the set of
  affected names varies by connection.

## Platform Notes

### FreeBSD / local_unbound

- Config: `/etc/unbound/unbound.conf`, generated by `local-unbound-setup`.
- The header warns that the file is regenerated; in practice manual edits
  persist until you re-run `local-unbound-setup`, which overwrites them.
- After config edits: `service local_unbound reload` for forwarder changes,
  but `restart` for `module-config`/trust-anchor changes (a SIGHUP reload does
  not reliably re-read those).

### Linux / unbound

- Local config lives under `/etc/unbound/unbound.conf.d/`.
- DNSSEC is usually enabled by default; the same fix applies — drop the
  validator module.

## How This Relates to vpn-switch

- vpn-switch configures the forwarder for the active interface through
  `resolvconf` and reloads `local_unbound` (see the connect DNS phase).
  Note a known wrinkle: the connect phase issues a `reload`, while the bundled
  `wg-resolvconf-up` helper (`template/phase/base/dns/resolvconf/`) uses a
  `restart` because a SIGHUP reload does not always re-read the resolvconf
  forwarder. If a forwarder change does not take effect, restart the service
  manually.
- vpn-switch does **not** touch DNSSEC settings; validation policy is left
  entirely to your resolver configuration (the change above is yours to make).
- DNS backends are pluggable: `resolvconf` (default), `unbound`, `dnsmasq`,
  `djbdns`, and `none` live under `template/phase/base/dns/`. The `unbound`
  backend writes its own forwarder to `/var/unbound/etc/unbound-vpn.conf` and
  calls `unbound-control reload`; choose `none` when DNS is managed externally.

## Quick Reference

```bash
# Diagnose
drill @10.2.0.1 example.com      # VPN DNS directly
drill @127.0.0.1 example.com     # local resolver (SERVFAIL => local resolver fault)
cat /var/unbound/forward.conf    # forwarder the resolver was given
sudo tcpdump -i wg0 -n port 53   # watch the queries

# Fix DNSSEC SERVFAIL (FreeBSD local_unbound)
sudo $EDITOR /etc/unbound/unbound.conf
#   comment out: auto-trust-anchor-file
#   add:         module-config: "iterator"
sudo service local_unbound restart

# Verify
ping example.com
curl https://ifconfig.me          # should show the VPN exit IP
```
